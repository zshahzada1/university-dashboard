Task: bb_client — expose bb_status from get_column_grade

Design Decisions

Extracted data.get("status") directly from the raw BB API response. No mapping of BB status strings at this layer — raw values are preserved for the backend service to interpret.

Deviations

None. Gemini followed the spec exactly.

Tradeoffs

Passing the raw BB status string (e.g. "NeedsGrading") rather than normalising it here keeps the client a thin wrapper. The backend grades service is the single place that interprets meaning.

Open Questions

None.

Task: Grades.tsx / Grades.module.css — replace dropdown with status badge

Design Decisions

Removed the <select> dropdown and all client-side Assignment state entirely. Status is now derived server-side and embedded in AssessmentGrade.status; the badge is purely presentational using s[a.status] CSS Module lookup. unmapped items suppress the badge but show a dashed "unmapped" label in the score column.

Deviations

None. Gemini (flash) followed the spec exactly. The .ungraded CSS class remains as dead code — it was present in the original CSS and is harmless.

Tradeoffs

Eliminating the /api/assignments call on the Grades page reduces load time and removes the two-source synchronisation problem. Manual submission status entry is no longer possible from the UI — any override must go through assignments.json directly, which is acceptable since overrides are rare (paper exams).

Open Questions

None.
